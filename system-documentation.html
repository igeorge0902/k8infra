Booking & Payment Flow

Booking · Hibernate & Caching Deep-Dive

The booking flow in mbooks-quarkus relies on a hand-managed Hibernate SessionFactory (not Quarkus-managed persistence) together with Infinispan-backed second-level and query caches. Understanding how these layers interact is essential for reasoning about data freshness, concurrency, and performance during the ticket-reservation lifecycle.

1. Singleton DAO & Shared Session

DAO is instantiated once via a double-checked DAO.instance() pattern. It holds:

  • A single SessionFactory built from hibernate.cfg.xml by HibernateUtil.
  • A static volatile Session object shared across all requests (opened at startup, reused via factory.getCurrentSession()).
  • All public methods are synchronized, serialising every database operation through the singleton — effectively making the DAO a global lock.

At construction time the factory's entire cache is evicted (factory.getCache().evictAllRegions()) to ensure a cold start. Statistics are enabled for monitoring (hibernate.generate_statistics=true).

3. Second-Level Cache (Infinispan)

Entity-Level Caching

Four entities are declared @Cacheable and mapped to named cache regions in hibernate.cfg.xml:

EntityCache RegionStrategyRationale
Moviemoviesread-writeMovie catalog changes infrequently; cache serves browse/search reads. Admin addScreen/updateScreen mutates movies, which invalidates the region entry.
Venuesvenuesread-writeVenue lookups (per movie, per location) are read-heavy during seat selection.
Locationlocationread-writeStatic venue-location data; almost never mutated after seed.
Ticketticketread-writeTickets are read when fetching purchase history; the read-write strategy ensures newly created tickets become visible immediately after commit.

Not cached: Seats, Screen, ScreeningDates, Purchase. Seat availability is the most latency-sensitive data in the booking flow and is deliberately not entity-cached to avoid serving stale "available" data.

Query-Level Caching

The Hibernate query cache (use_query_cache=true) is enabled and used extensively via .setCacheable(true) with named cache regions:

Query / DAO MethodCache RegionBehaviour
getAllMovies()moviesFull movie list (with optional category filter & paging) is query-cached. CacheMode.NORMAL reads from and writes to cache.
searchMovies()moviesSearch-by-name results share the same movies region; any movie entity change auto-invalidates.
getAllVenuesForUpdate()moviesAdmin venue+movie join query; cached in the movies region because the driving entity is Movie.
getVenuesForMovie() / getVenueByLocation()venuesVenue lists for a given movie/location are cached; invalidated when a Venue entity is updated.
getAllLocations() / getLocationForVenue() / getLocationForMovie()locationLocation queries are cached; effectively static after initial data load.
getPurchases(uuid)purchasesPer-user purchase list is query-cached. New bookings create new Purchase rows, which invalidate the cache for subsequent reads.
getTicketPerPurchase()(default)Ticket-per-purchase query uses setCacheable(true) without an explicit region; falls back to the default query cache.

Critical design choice: Seat-availability queries (getSeatsForScreening, getSeatForAvailability) are not query-cached. These always hit the database to guarantee the freshest reservation state.

5. Booking Transaction — Step-by-Step with Cache Effects

The primary booking entry point is BookController.fullcheckout2() → TicketService.reserveTickets() → BookingHandlerImpl.returnTickets() → DAO.bookTickets().

  1. Create Purchase row — DAO.setPurchaseId(uuid, orderId) inserts a new Purchase entity with a generated ID. Because Purchase is not entity-cached, this does not touch the L2 cache. However, it does invalidate the purchases query cache region (timestamp update).
  2. Lock each requested seat — For every seat number, DAO.getSeatForAvailability(screeningDatesId, seatNumber) executes an HQL join with LockModeType.PESSIMISTIC_WRITE. This acquires a database-level SELECT … FOR UPDATE row lock on the Seats row, preventing concurrent bookings from reserving the same seat. Seats is not L2-cached, so the lock always hits the database.
  3. Availability check — If any seat's isReserved != "0", a CustomExceptions is thrown, aborting the entire booking. The transaction is not committed, so the pessimistic locks are released on rollback.
  4. Create tickets & mark seats reserved — For each available seat:
    • A new Ticket entity is created linking the seat, screen, purchase, price, and tax.
    • The seat's isReserved is set to "1".
    • session.save(ticket) + session.update(seat) are called — these are batched by Hibernate (batch size 20).
    Since Ticket is L2-cached (read-write), the newly persisted ticket is placed into the ticket cache region upon transaction commit. Seats is not cached, so its update goes directly to the database.
  5. Return tickets (no explicit commit in bookTickets) — The @Transactional annotation on bookTickets() manages the commit. On success, the Hibernate flush writes all batched inserts/updates, the L2 cache is updated for Ticket entities, and query-cache timestamps for affected regions are bumped.

6. Post-Booking Cache Interactions

ActionCache Effect
Seat map refresh (DAO.updatetedSeats())Loads ScreeningDates → Venues → Screen → Seats from database (no cache). Returns the live reservation state to the client.
Purchase history (DAO.getPurchases(uuid))Query-cached in purchases region. After a new booking the timestamp cache invalidates stale results, so the next call re-queries the database and caches the updated list.
Ticket-per-purchase (DAO.getTicketPerPurchase())Uses JOIN FETCH to eagerly load Screen → Movie and ScreeningDates → Venues in one query. Result is query-cached (default region). The joined Movie and Venues entities populate the L2 entity cache.
Cancel ticket (DAO.cancelTicket())Loads and deletes Ticket (evicts from ticket L2 region), resets Seats.isReserved="0" (direct DB update). If all tickets are cancelled the Purchase is deleted. Query caches are invalidated by timestamp.
Delete purchase (DAO.deletePurchase())Bulk-fetches tickets with LEFT JOIN FETCH, resets seats in batches of 20 (with session.flush()/clear() to avoid memory pressure), then bulk-deletes the purchase via HQL. The flush/clear cycle means intermediate entity state is not held in the persistence context.

7. Admin Screen Management & Cache

Admin operations (addScreen, updateScreen, deleteScreen) modify Movie, Venues, Screen, ScreeningDates, and Seats entities:

  • addScreen — DAO.insertNewScreen() opens a separate session (factory.openSession()) rather than using the singleton shared session, to avoid deadlocking with concurrent reads. It creates Location lookup → Movie update (sets category) → new Venue → new ScreeningDates → new Screen → batch-generates Seats (row×col). After commit, the movies and venues L2 entity caches are invalidated for the affected entities. A Kafka message is published via KafkaMessageProducer on a background executor.
  • updateScreen — Uses the shared session. Updates Location on Venue, ScreeningDate, Movie category, and Screen→Movie link. Commit invalidates movies, venues, and location cache regions for touched entities.
  • deleteScreen — Deletes a ScreeningDates entity (cascades to Venues/Screen/Seats via CascadeType.ALL). Evicts related entries from all affected L2 cache regions.

8. Concurrency Model & Cache Consistency

How Consistency is Maintained

  • Database-level pessimistic locks — PESSIMISTIC_WRITE on seats during booking prevents double-selling. This is the authoritative concurrency control.
  • Synchronized DAO — The singleton's synchronized methods serialise all Hibernate operations, preventing session-level race conditions (Hibernate sessions are not thread-safe).
  • Read-write L2 cache strategy — Infinispan's read-write strategy uses soft locks: when an entity is updated, the cache entry is replaced with a lock marker until the transaction commits. Concurrent reads during this window bypass the cache and hit the database.
  • Timestamp-based query cache invalidation — The timestamps Infinispan cache tracks per-table modification times. When a booking commits (touching Ticket, Seats, Purchase tables), all query-cache entries that reference those tables are invalidated.

Known Limitations

  • The global synchronized lock on the DAO serialises all operations — reads and writes alike. This limits throughput to one concurrent database operation at a time. Movie browsing (read-only, cacheable) is blocked while a booking transaction holds the lock.
  • The static shared session means all operations share the same persistence context. Long-running admin operations (e.g. generating 100+ seats) can grow the persistence context, increasing memory pressure until the next flush()/clear().
  • Seats is not L2-cached (by design, for freshness), but it also means repeated seat-map loads (e.g. polling for availability) always hit the database.
  • The 5-minute idle timeout on entity and query caches means that after a quiet period, the first movie-listing request will see a cache miss and full DB round-trip.

Registration & Voucher Flow

Voucher State Machine

The voucher lifecycle is managed via the voucher_states table with three states:

StateMeaningTransition
freeVoucher is available for useCreated when admin inserts voucher; get_voucher() verifies this state.
processingVoucher claimed, activation pendingset_voucher() during registration. User must enter the activation code (token2 from email).
registeredVoucher fully activatedregister_voucher() → activate_voucher() after successful email code entry. isActivated flag set to 1.

Rollback: If registration fails after set_voucher, the stored procedure reset_voucher() resets the voucher to free and deletes the partially-created user and voucher records.

isActivated check: On login, HelloWorld calls isActivated(user). If the user's voucher is not yet activated, login returns HTTP 300 with "activation": "0", prompting the client to show the activation screen.

Password Reset Flow

Three-Step Flow

  1. Request (POST /login/ChangePassword) — Client sends email. The servlet calls find_email() to verify the email exists, then forgot_password(email, time) which inserts/updates the forgotPsw table. The forgotPsw_BEFORE_INSERT trigger auto-generates a random forgotRequestToken. An email is sent via SendHtmlEmail with the token.
  2. Verify Code (POST /login/ChangePasswordCode) — Protected by ChangePswFilter. Client sends email + code (SHA512-hashed). The servlet calls find_email2(email) to retrieve the stored token + timestamp. If the code matches and the request time is within the expiry window (configured via TIME context param, default 8,640,000 ms = 2.4 hours), the flow proceeds.
  3. Set New Password (POST /login/ChangePasswordNewPassword) — Client sends email + code + new password hash (SHA3-512). The servlet calls update_password(pass, email) which updates the hash_ column and invalidates the reset token by setting isValid=0.

Security notes: The confirmation code is SHA512-hashed by the client before sending. The new password is SHA3-512-hashed (same as login). Both web and iOS clients use CryptoJS for these operations. The ChangePswFilter ensures steps 2 and 3 have a valid session context before proceeding.

HTTP Interceptor Chain

The AngularJS app registers a custom HTTP interceptor and request transformer in app.js:

  1. Request interceptor — Checks that localStorage.hmacSecret exists. Rejects unsigned requests. Sets X-URL header to the request URL for HMAC message construction.
  2. Request transformer — For every POST with a body:
    • Adds X-SESSION-TOKEN header (if available in localStorage)
    • Sets X-MICRO-TIME to current epoch milliseconds (replay protection)
    • Computes X-HMAC-HASH = HMAC-SHA512(url:body:microTime:contentLength, hmacSecret)
    • Sets X-URL to the browser-generated deviceId
  3. Response error handler — Catches 502 responses (session expired, forwarded from AuthFilter)

Login Handshake (Web Client)

The web login in LoginController mirrors the iOS HMAC flow:

  1. Hash password: SHA3(password, {outputLength: 512})
  2. Generate HMAC secret: HmacSHA512(username, encodeURIComponent(passwordHash)) → stored in localStorage.hmacSecret
  3. Generate deviceId: Browser fingerprint from navigator.mimeTypes.length + userAgent + plugins.length + screen.height + width + pixelDepth
  4. POST to /login/HelloWorld with body user=&pswrd=&deviceId=, headers: Content-Type: application/x-www-form-urlencoded, Authorization: Basic btoa(user:hash)
  5. On success: Store X-Token in localStorage, rotate HMAC secret to SHA512(X-Token, hmacSecret)
  6. Redirect: If session is "raked" → /login/main.html (movie listing). If mobile device detected → /login/tabularasa.html (stripped UI).

iOS Architecture & Networking

Networking Stack

URLSession Singleton

All HTTP requests go through URLSession.sharedCustomSession — a singleton configured with:

  • Cookie acceptance: .always (session cookies from dalogin are automatically forwarded)
  • HTTP pipelining: enabled for performance
  • Self-signed cert trust: CustomURLSessionDelegate trusts milo.crabdance.com, localhost, igeorge1982.local

GeneralRequestManager

Primary HTTP client used by all view controllers. Supports:

  • GET/POST with configurable headers, query params, body params
  • Realm-based response caching: if isCacheable == "1", responses are stored in Realm with 1-hour TTL
  • Returns SwiftyJSON.JSON or raw Data (for images)

HMAC-SHA512 Handshake (client side)

The iOS login exactly mirrors the Python test scripts:

  1. Secret: HMAC-SHA512(username, passwordHash)
  2. Message: /login/HelloWorld:user={u}&pswrd={h}&deviceId={d}:{microTime}:{contentLength}
  3. Hash: HMAC-SHA512(message, secret) → sent as X-HMAC-HASH header
  4. Also sends X-MICRO-TIME, Content-Length, M-Device (AES-encrypted deviceId)

Implementation: CryptoJS.swift bridges to hmac-sha512.js via JavaScriptCore. The same CryptoJS library provides AES encryption for the deviceId and XSRF token reconstruction.

Session Flow (iOS perspective)

  1. Login: LoginVC posts to /login/HelloWorld with HMAC headers. On 200, stores JSESSIONID and X-Token in UserDefaults; cookies are auto-stored by HTTPCookieStorage.
  2. Subsequent requests: RequestManager adds Ciphertext header (= X-Token from UserDefaults). Cookies (JSESSIONID, XSRF-TOKEN) are sent automatically by the shared URLSession.
  3. Protected endpoints: Calls to /login/admin, /login/GetAllPurchases, /login/CheckOut etc. pass through dalogin's AuthFilter which validates session + XSRF-TOKEN cookie.
  4. Logout: GET /login/logout invalidates the server session. Client sets ISLOGGEDIN = 0 in UserDefaults and clears cookies.

Refactoring Suggestions

🔴 HIGH — Thread-Safety: Remove Static Mutable State

Where: HelloWorld.java, SQLAccess.java, mbooks DAO.java, all filters

Problem: Servlet fields like pass, user, session, token2 are declared volatile static. Under concurrent requests, these get overwritten by other threads between reads. The synchronized on doPost() serializes all requests through a single servlet instance, destroying throughput.

Fix: Move all per-request state to local variables inside the method. The servlet should be stateless. Remove synchronized from request methods. Same applies to SQLAccess — the static fields (hash, token, uuid, rs, etc.) should be local to each method call.

// BEFORE (broken under concurrency)
private volatile static String user;
public synchronized void doPost(...) { user = request.getParameter("user"); ... }

// AFTER
public void doPost(...) { String user = request.getParameter("user"); ... }

🔴 HIGH — Database Connections: Use Connection Pooling

Where: DBConnectionManager.java, SQLAccess.java

Problem: Every SQLAccess method calls dbManager.getConnection() which creates a new raw DriverManager.getConnection() each time. No connection pooling means every login/query creates and tears down a TCP connection to MySQL.

Fix: Use Quarkus's built-in Agroal connection pool (quarkus-agroal + quarkus-jdbc-mysql). Inject javax.sql.DataSource and let Quarkus manage pooling. The QUARKUS_DATASOURCE_* env vars are already configured in the K8s manifest.

@Inject DataSource dataSource;

public String getHash(String pass, String user) throws SQLException {
    try (Connection conn = dataSource.getConnection();
         CallableStatement cstmt = conn.prepareCall("{call get_hash(?, ?)}")) {
        cstmt.setString(1, pass);
        cstmt.setString(2, user);
        ResultSet rs = cstmt.executeQuery();
        return rs.next() ? rs.getString(1) : null;
    }
}

🔴 HIGH — mbooks DAO Singleton: Replace with CDI-Managed Sessions

Where: mbooks-quarkus DAO.java, HibernateUtil.java

Problem: DAO.instance() is a hand-rolled singleton that opens a Session once and reuses it forever. This is inherently thread-unsafe, causes stale data, and leaks sessions. All methods are synchronized, serializing all database access to a single thread.

Fix: Follow the pattern already used in mbook-quarkus — inject a Quarkus-managed Session into a @RequestScoped DAO. Let the container manage session lifecycle, transactions, and connection pooling.

🔴 HIGH — Hardcoded Credentials in Source

Where: BookController.java (Braintree keys), CookieFilter.java / HelloWorld.java (AES SALT/IV), SQL files

Problem: Braintree merchant credentials, AES encryption constants, and MySQL passwords are committed in source code.

Fix: Move all secrets to Kubernetes Secrets or env-var overrides. Reference via @ConfigProperty or System.getenv(). The AES SALT/IV must be consistent between dalogin and mbook — use a shared K8s Secret.

🟡 MEDIUM — Replace Stored Procedures with Application Logic

Where: mysql_8/login.sql, SQLAccess.java

Problem: 24 stored procedures contain business logic (voucher state machines, token management, user registration). They're hard to version, test, debug, and have caused the login vs login_ breakage. Trigger-based auto-generation of UUIDs and tokens is invisible and hard to reason about.

Fix: Gradually move logic into Java service classes. Keep triggers only for truly database-level concerns (e.g., audit timestamps). Use application-generated UUIDs (UUID.randomUUID()) and tokens (SecureRandom).

🟡 MEDIUM — Extract a Shared Security Module

Where: dalogin filters/, mbook filters/, mbooks filters/

Problem: AES constants (SALT, IV, KEYSIZE, ITERATIONCOUNT) are duplicated across 6+ classes in 3 services. The XSRF token verification logic is implemented independently in AuthFilter, CookieFilter, and HelloWorld.

Fix: Create a shared cinemas-security Maven module with the AES/HMAC utilities and constants. All three services depend on it. Consider moving to a standard JWT/OAuth2 approach long-term.

🟡 MEDIUM — API Gateway Pattern: Reduce dalogin Proxy Servlets

Where: CheckOut.java, GetAllPurchases.java, ManagePurchases.java

Problem: dalogin acts as a manual API gateway — each proxy servlet does nothing but forward the request to mbooks via ServiceClient, adding session headers. This creates unnecessary hops and maintenance burden.

Fix: Let Apache (or a proper API gateway) handle header injection. Move session-header forwarding to an Apache mod_headers config or a NGINX auth_request sub-module. Clients can call mbooks directly for non-session-sensitive endpoints.

🟡 MEDIUM — mbooks BookController: God-Class Decomposition

Where: mbooks-quarkus BookController.java (1171 lines)

Problem: Single controller handling movie browsing, seat selection, payment, purchase management, and search — with inline JSON assembly and direct Braintree SDK calls.

Fix: Split into domain-specific resources: MovieResource, BookingResource, PaymentResource, PurchaseResource. Use DTO classes instead of manual JSONObject construction (Jackson will auto-serialize).

🟢 LOW — Persistent Pictures Volume

Where: quarkus-backend.yaml

Problem: Pictures volume is emptyDir — lost on every pod restart. Requires manual kubectl cp after each deployment.

Fix: Use a PersistentVolumeClaim, or build the pictures into the Docker image, or use a Minikube hostPath mount. A ConfigMap-based init container could also sync from a git-hosted pictures repo.

🟢 LOW — Standardize Error Responses

Where: All controllers and servlets

Problem: Error responses use inconsistent formats: {"Error Message": "..."}, {"Error Details": {...}}, {"error": "..."}, raw strings. HTTP status codes are inconsistent (502 for auth failures, 300 for activation required).

Fix: Define a standard error envelope: {"error": {"code": "AUTH_FAILED", "message": "..."}}. Use proper HTTP status codes (401 for auth, 403 for forbidden, 409 for conflict).

🟢 LOW — Add Tests

Where: All services

Problem: Test coverage is minimal. Only two Python integration tests exist. No unit tests for business logic, security filters, or stored procedures.

Fix: Add Quarkus @QuarkusTest integration tests for each REST endpoint. Add unit tests for hmac512, AesUtil, TicketService, PaymentService. Add Testcontainers-based tests for stored procedures. The Python tests are a good starting point — expand them to cover registration, booking, and error paths.

Glossary

TermDefinition
daloginAuthentication and session gateway service. Handles login, registration, session management, and proxies protected requests to downstream services. Name derived from "Da Login".
mbookUser/device/profile microservice. Serves user data, device lists, news events, and realtime notifications. Name derived from "My Book" (user book).
mbooksMovie/booking/payment microservice. Handles the movie catalog, seat reservation, Braintree payment processing, and purchase history. Name derived from "My Books" (movie bookings).
HMAC-SHA512Hash-based Message Authentication Code using SHA-512. Used for the login handshake — the client proves knowledge of the password hash without transmitting it in plaintext. The message includes the URL, body, timestamp, and content-length.
XSRF-TOKENCross-Site Request Forgery token. An AES-encrypted combination of TIME_ (session timestamp) and token2 (device-specific random token). Set as a cookie by dalogin and validated on every protected request by AuthFilter and CookieFilter.
CiphertextRequest header that must match the token2 header. Used by mbook's CiphertextFilter as a secondary verification that the request was forwarded by dalogin's ServiceClient (which copies X-Token to both headers).
token1UUID generated by the Tokens_BEFORE_INSERT trigger when a device logs in. Used to identify the session in GET /rest/user/{user}/{token1}.
token2Random value generated by the Tokens_BEFORE_INSERT trigger. Used for XSRF-TOKEN encryption and inter-service authentication (forwarded as the Ciphertext and token2 headers).
TIME_Timestamp stored in the Tokens table when a device logs in. Used as input to AES encryption for XSRF-TOKEN generation. Header name preserved as-is for client compatibility.
ServiceClientRESTEasy-based HTTP client in dalogin that proxies requests to mbook/mbooks. Registers RequestFilter to forward session-derived headers (X-Token, Ciphertext, cookies, uuid, token2, TIME_).
Singleton DAOThe DAO class in mbooks-quarkus uses a hand-rolled singleton pattern (DAO.instance()) with a static Hibernate Session. All methods are synchronized, serialising database operations.
L2 CacheHibernate's second-level cache, backed by Infinispan. Caches entity instances (Movie, Venues, Location, Ticket) across Hibernate sessions. Reduces database round-trips for frequently-read data.
Query CacheHibernate's query result cache. Stores the IDs returned by HQL queries. Works in conjunction with the L2 entity cache and timestamps cache for invalidation.
Pessimistic Write LockLockModeType.PESSIMISTIC_WRITE → SELECT ... FOR UPDATE. Used during seat reservation to prevent double-selling. Holds a database-level row lock until the transaction commits or rolls back.
BraintreePayment gateway by PayPal. The system uses Braintree Sandbox for development. PaymentService generates client tokens and processes transactions via the Braintree Java SDK.
Linear TemplateThe original HTML/CSS responsive template used for the web frontend. Provides layout variants (left-sidebar, right-sidebar, no-sidebar) and a clean design.
login_The MySQL database schema for authentication, user, device, token, voucher, and password-reset data. The trailing underscore is intentional — a legacy database named login (without underscore) must not exist.
bookThe MySQL database schema for movies, venues, screens, screening dates, seats, tickets, and purchases.
Apache Reverse Proxyhttpd 2.4 running inside Kubernetes that routes HTTP requests to the correct backend service based on URL prefix (/login, /mbook-1, /mbooks-1, /simple-service-webapp). Also handles WebSocket upgrade proxying.
NGINX IngressKubernetes Ingress controller that terminates TLS (using the cinemas-tls secret) and forwards traffic to the Apache reverse proxy. External entry point for all client traffic.
PVCPersistent Volume Claim. The pictures-pvc provides 500Mi of storage for movie posters, venue photos, and profile pictures served by simple-service-webapp.
fix-sprocs.sqlPost-import fix script that re-creates stored procedures with correct login_. table references (replacing any legacy login. references) and adds missing columns.
fix-triggers.sqlPost-import fix script that re-creates triggers with correct table-name casing for MySQL 8 on Linux (case-sensitive filesystem).
